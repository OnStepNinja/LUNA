# LUNA — AI シリアルブリッジ ガイド
## RS-232C機器をAIで自律制御する、新しいパラダイム

**対象**: 技術者・エンジニア
**バージョン**: LUNA (AiBridgeMCP v3.9.1)
**リビジョン**: r1.0
**最終更新**: 2026-03-19
**作成日**: 2026-03-19

---

## はじめに

LUNAは、ESP32とClaude AIをつなぐブリッジとして開発されました。
しかし、その本質はもっと広い可能性を持っています。

> **RS-232Cポートを持つあらゆる機器を、AIが直接制御できる。**
> **さらに、Luaスクリプトが Claude 不在でも自律的に動き続ける。**

工場の生産設備、研究室の計測器、医療機器、天文機器——
これまで専用ソフトウェアや専門知識が必要だったあらゆる機器が、
**「説明書を渡すだけ」でAIシステムになります。**

## 2. 「説明書を渡すだけ」の衝撃

### なぜマニュアルだけで動くのか

AIは膨大な言語知識を持っています。
RS-232Cのコマンド体系、シリアル通信プロトコル、各種機器の制御方法——
これらは文字情報として記述できます。

つまり、機器のマニュアルをAIに渡せば：

```
マニュアルに書いてあること:
  "MEAS:VOLT:DC?" → 直流電圧を測定するコマンド
  応答形式: "+1.2345E+00\r\n"

AIが理解すること:
  「直流電圧を測りたいときは MEAS:VOLT:DC? を送信する」
  「応答は指数表記の数値で返ってくる」

ユーザーが言うこと:
  「直流電圧を測って」

AIがすること:
  MEAS:VOLT:DC? をシリアルポートへ送信 → 応答を受信 → 結果を解釈して報告
```

### 数ヶ月の開発が、数分になる

| | 従来方式 | LUNA方式 |
|---|---|---|
| 準備時間 | 数週間〜数ヶ月 | 数分（マニュアルを渡すだけ） |
| 必要なスキル | プログラミング、通信プロトコル知識 | 自然言語で話せること |
| 対応機器変更時 | 新たに開発が必要 | 新しいマニュアルを渡すだけ |
| 担当者不在時 | 使えなくなる | マニュアルがあれば誰でも使える |
| コスト | 高（開発費） | 低（ハードウェアのみ） |
| **夜間・無人運転** | **別途システムが必要** | **Lua スクリプトで自律実行** |

### 各コンポーネントの役割

| コンポーネント | 役割 |
|---|---|
| **Claude Desktop** | AIの頭脳。マニュアルを理解し、制御戦略を決定する |
| **MCP（Model Context Protocol）** | AIとLUNAの標準通信プロトコル（Streamable HTTP） |
| **ESP32 (LUNA)** | WiFi↔RS-232Cの変換ブリッジ + Lua自律実行エンジン |
| **Lua VM** | Claude不在でも動く自律制御スクリプトエンジン（v3.9.1新機能） |
| **RS-232C** | 機器との物理的な通信路（40年以上の標準規格） |
| **機器マニュアル** | AIへの「取扱説明書」。これがあれば制御可能 |

### なぜESP32か

```
ESP32を選んだ理由:
  ✅ WiFi内蔵      → ケーブル不要、どこにでも設置可能
  ✅ 安価（数百円） → 機器ごとに1台置いても低コスト
  ✅ RS-232C対応   → MAX3232等で簡単に接続可能
  ✅ SPIFFS内蔵    → 設定・データ・Luaスクリプトの永続保存
  ✅ 入手容易      → 世界中で購入可能
  ✅ 実績ある安定性 → 産業用途にも耐える
  ✅ Lua VM搭載可能 → 25KB RAM で Lua 5.1 が動作する
```

### なぜMCPか

```
MCPを選んだ理由:
  ✅ Anthropic公式の標準プロトコル
  ✅ Claude Desktopとネイティブ統合
  ✅ 将来的に他のAIにも対応可能
  ✅ ツール（Tool）として機能を拡張可能
  ✅ Streamable HTTPでリアルタイム応答（v3.9.1）
  ✅ prompts/list・prompts/get でLUNA使用法を自動通知
```

## 5. ハードウェア接続

### 必要なもの

```
① LUNA（ESP32搭載ボード）              ← 本製品
② RS-232Cケーブル（クロス or ストレート、機器に合わせて）
③ 電源（USB 5V）
④ WiFiルーター（既存のもので可）
⑤ Claude Desktopがインストールされたパソコン
⑥ 制御したい機器のマニュアル（PDF等）
```

### 配線

```
【LUNA】                    【RS-232C機器】

  TX2 (GPIO17) ─────────────► RXD
  RX2 (GPIO16) ◄───────────── TXD
  GND          ─────────────── GND

  ※ MAX3232等のRS-232Cレベル変換回路を経由して接続
  ※ フロー制御が必要な場合は RTS/CTS も接続
```

### ボーレート設定

| エディション | 設定 |
|---|---|
| **LUNA（Free）** | 9600 bps 固定（8N1） |
| **SerialBridge Pro（v4.2.0）** | serial_configツールで変更可能 |

機器のボーレートが異なる場合はPro版をご検討ください。

## 6. AIに機器を制御させる手順

### Step 1: LUNAの初期設定

LUNAの電源を入れ、WiFiとLocation Nameを設定します。
詳細は **LUNA_README.md** を参照してください。

```
設定完了後の確認:
  WiFi SSID: AiBridgeMCP_Ai-{LocationName}
  MCP URL:   http://{IPアドレス}/mcp
```

### Step 2: Claude DesktopにMCPサーバーを登録

Claude Desktopの設定ファイル（`claude_desktop_config.json`）に追加します。

```json
{
  "mcpServers": {
    "LUNA-Lab": {
      "command": "npx",
      "args": [
        "-y",
        "mcp-remote",
        "http://192.168.1.100/mcp"
      ]
    }
  }
}
```

> **注意**: v1.8.8 以前の `/mcp/sse` は旧形式です。LUNAは `/mcp`（Streamable HTTP）を使用します。

### Step 3: マニュアルをAIに渡す

Claude Desktopのチャット画面で、機器のマニュアルを渡します。

```
方法A: PDFをドラッグ&ドロップ
  → Claude Desktopにマニュアル(PDF)をそのまま渡す

方法B: テキストで貼り付け
  → マニュアルのコマンド一覧部分をコピー&ペーストする

方法C: 要点をテキストで説明
  → 「この機器はRS-232C接続で、コマンドは...」と説明する
```

**ポイント**: コマンド一覧、応答フォーマット、初期化手順が含まれていると最も効果的です。

> **LUNA の場合**: 接続時に `prompts/get luna_guide` が自動実行され、ClaudeがLUNAの使い方（serial.*ツール、Luaスクリプト等）を自動的に把握します。

### Step 4: 自然言語で指示する

あとは普通の会話でAIに指示するだけです。

```
例:
  ユーザー: 「直流電圧を測定して」
  AI:       MEAS:VOLT:DC? を送信
             → 応答受信 → 「直流電圧は 1.2345V です」

  ユーザー: 「1秒ごとに10回測定して、平均を出して」
  AI:       10回連続測定 → データ収集 → 平均値を計算して報告

  ユーザー: 「電圧が規格外（1.0V未満または1.5V超）なら教えて」
  AI:       測定 → 判定 → 異常時のみアラート
```

### Step 5（LUNA新機能）: Luaスクリプトで自律制御

Claudeに頼まず、ESP32上のLuaスクリプトが単独で動き続けます。

```
例:
  ユーザー: 「1分ごとに電圧を測って、異常なら通知するスクリプトを作って」
  AI:       Luaスクリプトを生成 → lua_load で保存 → lua_run で実行開始
            → 以降はClaude不在でもESP32が自律監視し続ける
            → 異常時は notify.set() でClaudeに通知
```

```
  最大値: 1.3456 V（3回目）
  最小値: 1.2201 V（7回目）
  平均値: 1.2890 V
  標準偏差: 0.0312 V
```

### 例B: Luaスクリプトによる自律監視（LUNA v3.9.1 新機能）

**シナリオ**: Claude不在で電圧を定期監視し、異常時に通知

**Claudeへの指示例**:
```
電圧を30秒ごとに測定して、1.0V未満または1.5V超なら
私に通知するLuaスクリプトをLUNAに書き込んで実行してください。
```

**AIが生成・保存するLuaスクリプト**:
```lua
-- 電圧監視スクリプト（voltage_monitor）
for i = 1, 20 do
  local res = serial.query("MEAS:VOLT:DC?\r\n", 1000)
  local v = tonumber(res)
  if v then
    log.write(string.format("%.4fV", v))
    if v < 1.0 or v > 1.5 then
      notify.set("電圧異常: " .. string.format("%.4f", v) .. "V")
      break
    end
  end
  hw.delay(30000)
end
```

**動作フロー**:
```
lua_load("voltage_monitor", <スクリプト>) → SPIFFS に保存
lua_run("voltage_monitor")                → 実行開始
                                          → Claude は離席してOK
（異常発生時）
notify.set() が発火
  → Claude が check_notify で検知
  → 「電圧異常: 0.8234V」と報告
```

### 例C: luna.run() によるスクリプトチェーン（LUNA v3.9.1 新機能）

**シナリオ**: 初期化→測定→レポートを3本のスクリプトで自律実行

```lua
-- スクリプト1: init（初期化）
serial.query("*RST\r\n", 2000)
serial.query("CONF:VOLT:DC\r\n", 500)
log.write("init_ok")
luna.run("measure")   ← 次のスクリプトを自律起動して終了
```

```lua
-- スクリプト2: measure（測定）
local results = {}
for i = 1, 10 do
  local r = serial.query("MEAS:VOLT:DC?\r\n", 1000)
  table.insert(results, r)
  hw.delay(1000)
end
log.write(table.concat(results, ","))
log.save()            ← SPIFFS に保存（レポートスクリプトへ引き渡し）
luna.run("report")    ← 次のスクリプトへ
```

```lua
-- スクリプト3: report（集計・通知）
log.load()            ← 測定データを受け取る
local sum = 0
local n = 0
-- （集計処理）
notify.set(string.format("測定完了: 平均 %.4fV", sum/n))
```

## 8. ダイレクト制御 vs Lua自律制御

LUNAには2つの制御モードがあります。

### ダイレクト制御方式（AI側からの直接制御）

```
【用途】

  → マニュアル通りのシーケンス実行

【フロー】
  Claude Desktop でマニュアルを渡す
      ↓

      ↓

      ↓

【使用するMCPツール】

  serial_write  ← 送信のみ（応答不要の場合）

```

```
【用途】
```

## 9. トラブルシューティング

### 通信できない場合

```
確認項目:
  □ ボーレートが機器と一致しているか（LUNA Free版は9600固定）
  □ クロス/ストレートケーブルの選択は正しいか
  □ GNDは接続されているか
  □ 電圧レベル変換回路（MAX3232等）は正しく動作しているか
  □ フロー制御（RTS/CTS）が必要な機器ではないか
```

### コマンドを送ったが応答がない場合

```
確認項目:
  □ コマンドの末尾の改行コード（CR/LF/CR+LF）は正しいか
  □ 機器のエコーバック設定はどうなっているか
  □ コマンド送信後の待機時間は十分か（timeout_ms を増やす）
  □ 初期化コマンドを先に送る必要がないか
```

### タイムアウトが発生する場合

```
対策:
  □ 機器の応答時間を確認し、timeout_ms を調整する
  □ AIに「応答を○秒待ってから次のコマンドを送って」と指示する
  □ 複雑な処理の前に機器の応答速度をテストする
```

### WiFi接続が不安定な場合

```
確認項目:
  □ ESP32とルーターの距離・障害物
  □ MCP接続タイムアウト設定
  □ GPIO34の設定（ソフトリセット vs ハードリセット）
```

### Lua スクリプトが動かない場合

```
確認項目:
  □ lua_status で state を確認（running / idle / error）
  □ lua_log でスクリプト内 log.write() の出力を確認
  □ serial.query のタイムアウト（ms）が短すぎないか
  □ luna.run() チェーンが LUA_MAX_CHAIN(10) を超えていないか
  □ log.save() をループ内で呼んでいないか（フラッシュ寿命に注意）
```

### 🔌 シリアルブリッジ系（機器制御の主力）

#### `serial_query`（最重要ツール）

コマンドを送信し、指定時間待機してから応答を受信します。

| 項目 | 内容 |
|------|------|
| **パラメータ** | `content`（必須）: 送信するコマンド文字列 |
| | `timeout_ms`（省略可）: 応答待ち時間（ミリ秒）。デフォルト500ms |
| **戻り値** | 機器からの応答文字列 |
| **使いどき** | コマンドを送ったら機器が応答を返す、典型的なRS-232C通信 |

```
使用例:
  serial_query("MEAS:VOLT:DC?\r\n")
    → "+1.2345E+00\r\n" を返す

  serial_query("*IDN?\r\n", timeout_ms=1000)
    → "HEWLETT-PACKARD,34401A,..." を返す
```

**ポイント**: コマンド末尾の改行コード（`\r\n` または `\r`）は機器の仕様に合わせてください。

---

#### `serial_write`

シリアルポートにデータを**送信するだけ**のツールです。

| 項目 | 内容 |
|------|------|
| **パラメータ** | `content`（必須）: 送信する文字列 |
| **戻り値** | 送信バイト数 |
| **使いどき** | 応答不要の一方向送信（プログラム転送、設定コマンド送信など） |

---

#### `serial_read`

シリアルポートの受信バッファにたまっているデータを**読み出すだけ**のツールです。

| 項目 | 内容 |
|------|------|
| **パラメータ** | なし |
| **戻り値** | 受信バッファの内容（データがなければ空文字） |
| **使いどき** | 機器側から自発的に送られてくるデータを受け取る場合 |

**注意**: 通常は `serial_query` を使う方がシンプルです。

---

#### `serial_config`（SerialBridge Pro版のみ）

シリアル通信のボーレートとパリティを変更します。
**LUNA Free版では使用できません（エラーを返します）。**

| 項目 | 内容 |
|------|------|
| **パラメータ** | `baud`（必須）: ボーレート |
| | `parity`（省略可）: `N`=なし（デフォルト）、`E`=偶数、`O`=奇数 |
| **対応ボーレート** | 300 / 600 / 1200 / 2400 / 4800 / 9600 / 19200 / 38400 / 115200 |

---

### 🤖 Lua自律制御系（v3.9.1 LUNA 新機能）

#### `lua_exec`

Luaコードをその場で即時実行します。テストや1回限りの処理に使います。

| 項目 | 内容 |
|------|------|
| **パラメータ** | `code`（必須）: Lua 5.1 のソースコード |
| **戻り値** | 実行開始確認（結果は lua_status で取得） |
| **使いどき** | 動作確認・テスト・簡単な1回実行 |

```lua
-- 使用例
lua_exec("serial.query('*IDN?\\r\\n', 1000)")
```

---

#### `lua_load`

LuaスクリプトをSPIFFSに保存します。電源断後も残ります。

| 項目 | 内容 |
|------|------|
| **パラメータ** | `name`（必須）: スクリプト名（英数字・_・-のみ、22文字以内） |
| | `code`（必須）: Lua 5.1 のソースコード |
| **保存先** | `/lua_{name}.lua` |

---

#### `lua_run`

保存済みスクリプトを実行します。

| 項目 | 内容 |
|------|------|
| **パラメータ** | `name`（必須）: lua_load で保存したスクリプト名 |
| **使いどき** | 定期実行・本番運用 |

---

#### `lua_status`

Lua VMの実行状態を確認します。

| 項目 | 内容 |
|------|------|
| **戻り値** | state（running/idle）、last_result、free_heap、uptime |
| | chain_index、chain_script（luna.run() チェーン進捗） |

---

#### `lua_stop`

実行中のLuaタスクを強制停止します。

---

#### `lua_log`

スクリプト内の `log.write()` で書かれたログを取得します（直近20行のリングバッファ）。

---

#### `lua_list`

SPIFFSに保存されているLuaスクリプト一覧を表示します。

---

### ツール選択チートシート

```
【機器にコマンドを送って応答を受け取りたい】
  → serial_query（timeout_msを機器に合わせて調整）

【プログラムや設定を一方的に流し込みたい】
  → serial_write（1行ずつ繰り返す）

【機器が自発的に送ってくるデータを受け取りたい】
  → serial_read

【LUNAが正常動作しているか確認したい】
  → get_system_info または get_network_status

【Luaスクリプトをテストしたい】
  → lua_exec

【Luaスクリプトを本番保存・実行したい】
  → lua_load → lua_run

【Luaスクリプトの実行状況を確認したい】
  → lua_status

【Luaスクリプトのログを確認したい】
  → lua_log

【Luaスクリプトから完了・異常をClaudeに通知したい】
  → スクリプト内で notify.set() → Claudeが check_notify で検知
```

## 付録B: Lua APIリファレンス（LUNA v3.9.1）

LuaスクリプトからESP32の機能を使うためのAPIです。

### serial モジュール

| API | 説明 |
|-----|------|
| `serial.query(cmd, timeout_ms)` | コマンド送信→待機→受信。主力API |
| `serial.write(data)` | 送信のみ |
| `serial.read()` | 受信バッファ読み出し |

### log モジュール

| API | 説明 |
|-----|------|
| `log.write(msg)` | ログバッファに書き込む（Claude が lua_log で読む） |
| `log.read([n])` | ログバッファを読む（スクリプト間データ受け渡し） |
| `log.save()` | ログを `/luna_log.txt` にSPIFFS保存（ループ内禁止） |
| `log.load()` | SPIFFSから読み込みバッファに展開（呼び出し時にバッファクリア） |

### hw モジュール

| API | 説明 |
|-----|------|
| `hw.delay(ms)` | 指定ミリ秒待機 |
| `hw.millis()` | 起動からの経過時間（ms） |
| `hw.reset()` | ESP32を再起動 |

### notify モジュール

| API | 説明 |
|-----|------|
| `notify.set(msg)` | Claude への通知メッセージをセット。check_notify で検知される |

### luna モジュール

| API | 説明 |
|-----|------|
| `luna.run(name)` | 次スクリプトをキューして現在スクリプトを終了（チェーン最大10本） |

## 付録E: RS-232Cレベル変換回路について

ESP32のGPIOは3.3V、RS-232Cは±12V系です。必ずレベル変換が必要です。

```
【推奨IC】
  MAX3232（3.3V動作、+/-15kV ESD保護）
  MAX232（5V動作、外付けコンデンサ4個必要）

【基本接続】
  ESP32 TX2(GPIO17) → MAX3232 T1IN → T1OUT → 機器 RXD
  ESP32 RX2(GPIO16) → MAX3232 R1OUT ← R1IN ← 機器 TXD
  ESP32 GND         → MAX3232 GND   ────────── 機器 GND
```
